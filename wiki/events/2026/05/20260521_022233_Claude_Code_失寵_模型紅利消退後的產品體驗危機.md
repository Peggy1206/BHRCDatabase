## Summary

Claude Code 曾是開發者工具的白月光，但因模型降智、額度限制不透明、上下文管理低效等問題，被 Codex 快速追上並超越。當模型能力趨同後，產品體驗和系統可靠性成為決定勝負的關鍵因素。

## Structured Analysis

## Claude Code 的衰落軌跡

### 1. 產品降智的三重打擊
- **推理強度下調**（3月4日）：默認推理從 high 改為 medium
- **快取 Bug**（3月26日）：歷史推理每次被清除，上下文斷裂
- **系統提示限制**（4月16日）：響應長度受限，編碼質量下降 3%
- **量化數據**（AMD AI總監 Stella Laurenzo 統計）：
  - 思考深度驟降 67%
  - 檔案讀取率下降 70%
  - 不良行為觸發次數飆升 173%

### 2. 額度機制的心理學陷阱
- **高峰時段加速消耗**：工作日太平洋時間上午 5-11 點，額度消耗加快
- **不可預測的成本**：同樣付費，但消耗速度不同
- **雙重收費模式**：先買月卡（$20），額度用完後還要額外付費
- **共享額度池**：Claude Code 和 Claude.ai 網頁版共用配額
- **被動漲價**：Tokenizer 變化導致同樣輸入消耗更多 token

### 3. 市場份額的急速轉換
- **npm 下載量對比**（截至5月3日）：
  - Codex：8610 萬次
  - Claude Code：720 萬次
  - 差距：12 倍（主要來自4月30日-5月3日的暴漲）

### 4. Codex 的產品優勢

#### 架構設計
- **雲端沙箱模式**：每個任務獨立環境運行，預載入倉庫
- **安全邊界清晰**：任務可平行執行，過程可追蹤
- **AGENTS.md 配置**：項目規則、測試命令、程式碼規範一份說明書

#### 功能迭代（2026年2月-4月）
- 2月2日：桌面應用發佈
- 2月5日：GPT-5.3-Codex 發佈，速度快 25%
- 4月16日：後台 computer use、內建瀏覽器、90+ 外掛生態
- 自動化功能：Issue 分類、CI/CD 監控、告警響應

#### Token 效率
- 同一 Express.js 重構任務：Codex 用 150 萬 token，Claude Code 用 620 萬 token（4倍差距）
- 原因：Codex 任務拆分平行化，Claude Code 易在同一上下文反覆探索

### 5. 兩款產品的本質差異

| 維度 | Claude Code | Codex |
|------|------------|-------|
| 定位 | 高級工程助理 | 完整生產力產品 |
| 環境 | 本地終端 | 雲端沙箱 |
| 靈活度 | 高（直接呼叫本地資源） | 中等（邊界清晰） |
| 安全性 | 依賴使用者管理 | 系統級保障 |
| 上下文管理 | 易污染、易燒 token | 隔離、高效 |
| 工程流程 | 共享終端感 | 任務派發感 |

### 6. 競爭進入新階段

**第一階段**：模型智力競爭
- Claude 因模型優勢獲得開發者寬容
- 產品缺陷被模型紅利遮蔽

**第二階段**：產品系統競爭
- 模型能力趨同後，體驗成為決策因素
- 額度透明性、穩定性、安全邊界成為企業級採購考量
- 個人開發者可憑感覺選擇，組織必須考慮風險

### 7. Claude Code 的翻身機會

**現有優勢**
- 模型底座仍強
- 開發者社區積累深厚
- 早期信任者基數大

**補齊方向**
- 權限管理透明化
- 多智能體協作
- 企業級部署能力
- 迭代節奏加快（已在進行中）

## 對人才招聘行業的啟示

### 工具選型決策
- 企業級工具採購不再只看功能，更看系統可靠性
- 成本透明性和可預測性成為關鍵評估指標
- 安全邊界和權限管理影響企業接入意願

### 人才市場影響
- AI 工具的穩定性直接影響開發者生產力和滿意度
- 工具切換成本低，但遷移決策反映市場信心轉移
- 企業在評估 AI 工具時會考慮長期可靠性

## Deepening Questions

- Claude Code 的衰落源於使用者無法預測成本和額度消耗，這反映了一個深層問題：當工具的『隱性成本』難以量化時，客戶會主動流失。在獵頭業務中，我們如何設計服務模式，讓客戶能夠清晰看見『每一次候選人推薦』『每一次面試評估』的實際投入成本，而不是籠統的月費或年費？這種成本透明化是否會改變客戶對我們服務價值的認知？
- Codex 的勝利不只是因為模型更強，而是它將『不確定性』轉化為『可控的工作流』。反觀獵頭行業，候選人匹配度、面試成功率、入職留任率等核心指標往往充滿不確定性。我們是否應該像 Codex 一樣，建立一套『可驗證、可重複、可預測』的服務流程框架，讓客戶能夠追蹤每個環節的成功率和失敗原因，從而從『相信獵頭的經驗』轉變為『相信系統的可靠性』？
